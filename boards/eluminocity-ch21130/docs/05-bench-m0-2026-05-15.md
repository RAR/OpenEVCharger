# Bench session — M0 plumbing PASS, M1 calibration BLOCKED on mains

Date: 2026-05-15
Unit: factory-virgin Eluminocity CH-21130 (Delta `EVMU3015HNSEL`), serial bench unit
Mains supplied during session: **120 V** (unit hardwired for 240 V split-phase)
Wi-Fi / network: unit on local LAN at 10.75.1.244, MQTT broker reachable at 10.75.1.54

## Headline

- **M0 (bridge plumbing) — PASS.** `delta-bridge` deployed to `/Storage`, attached
  read-only to SysV shmem `0x153E`, published MQTT Home Assistant discovery
  topics, entities appeared in HA, LWT honoured on SIGINT. Bridge architecture
  and the MQTT/shmem layer are validated against live hardware.
- **M1 (calibration / offset mapping) — BLOCKED.** The unit is in a latched
  undervoltage fault from being on 120 V, which disables the J1772 pilot
  generator (CP measured at −11.9 V at idle). Test plug A/B/C switching cannot
  produce CP-state transitions while the pilot is off, so several entities we
  hoped to ground-truth (cable state, pilot duty, VRMS, IRMS) cannot be
  characterised under current bench conditions.

Resume on a 240 V supply (or with the undervoltage fault otherwise cleared).

## What the live segment actually contains

`shmem_offsets.h` (and its source-of-truth, `decode_sharemem.py`) inherited
several offsets from inspecting a static dump (`mtd4-sharemem.bin`). Live
evidence from this bench session says many of them are wrong. The decoder
itself was honest about this — most of its labels carry `"?"` suffixes — but
the bridge inherited them without the hedge.

### `shmem` segment metadata (verified)

| | |
|---|---|
| Key | `0x153E` ✅ (matches `SHMEM_KEY`) |
| Size | `262144` bytes = 256 KiB ✅ (matches `SHMEM_SIZE`) |
| Perms | `777` |
| Owner | `vern` (same uid the bridge runs as) |
| `nattch` during session | **11** (one is our bridge; ten are stock daemons) |

`nattch=11` is the strongest evidence the producer side is fully alive — there
is *no* "is the segment being written?" question to answer; it is.

### Offset table — current `shmem_offsets.h` vs reality

| Symbol | Current offset | What's actually there | Verdict |
|---|---|---|---|
| `OFF_CONNECTOR_STATE` | `0x0a00` (u8) | Always `0x00`, didn't move during plug events | **Wrong** — try elsewhere |
| `OFF_FAULT_FLAGS`    | `0x0a07` (u8) | `0x03` resting; pulses to `0x02` during retry events | Plausible "latched fault" register; semantics unconfirmed |
| `OFF_HEARTBEAT`      | `0x0a08` (u8) | Resting `0x04`; pulses to `{0x01,0x02,0x03}` during polls | **Not a heartbeat** — likely "currently-polling task index" |
| `OFF_STM32_LINK`     | `0x0a0b` (u8) | Static `0x00` despite the link being unambiguously alive | **Wrong** — try elsewhere or semantics inverted |
| `OFF_VRMS`           | `0x0a10` (u8) | `0x32` = 50 — almost certainly **J1772 pilot duty %** (30 A → 50 %) | **Wrong** — drop, this is duty, not voltage |
| `OFF_IRMS`           | `0x0a24` (u8) | `0x1e` = 30 — sits adjacent to the netcfg strings → **rated/configured ampacity**, not measured I | **Wrong** — drop, this is config, not metering |
| `OFF_FW_UPGRADE_GATE` | `0x0a63` (u8) | `0x00`, didn't change | Plausible but unverified |
| `OFF_ALARM_BITMAP`   | `0x0138` (32 B) | `01 64 a8 c0 ...` — that's `192.168.100.1` LE, then zeros, with the same IP repeated at `0x0158` | **Wrong** — alarm bitmap is *not* at 0x0138; that region holds network config |

`decode_sharemem.py:38-58` agrees with this in hindsight — `0x100`-area is
explicitly network config (`UPDATE_IP flag`, IP, netmask, gateway, DNS), and
the alarm bitmap was a guess that the live segment doesn't support.

### Bytes that DO move (in 0x0000..0x2000)

Discovered by 90-second 10 Hz watch. With the test plug inserted and switched
A→B→C→B→A→unplug, **only these bytes ever changed**, and *none* of them took
sustained new values keyed to plug state:

| Offset | Behaviour | Likely meaning |
|---|---|---|
| `0x0000` | u8 oscillates `0xa0..0xd0`, ~1 Hz | ADC channel (probably CP-sense raw or temp) |
| `0x0004` | u8, slow flip `0x07 ↔ 0x08` | Another ADC channel |
| `0x000c` | u8 oscillates `0x5f..0x67` | Another ADC channel |
| `0x0366..0x0370` | 10-byte ASCII buffer rewritten on each "event burst" | Log/event string ("…0-11.760" toggling with "12.0.0.000…") |
| `0x0a01` | pulses `1 → 2 → 1` during every "event burst" | Producer "work in progress" flag, **not cable state** |
| `0x0a07` | pulses `3 → 2 → 3` during *some* bursts | "Interesting event" flag during the burst |
| `0x0a08` | pulses `4 → {1,2,3} → 4` during bursts | Polling-task-index or context byte |

"Event bursts" happened at irregular intervals (8.5, 6.9, 2.4, 32.6, 4.7, 23.8 s
within the 90 s window), so they're event-driven, not periodic. Hypothesis:
they're undervoltage-retry attempts by `/root/main`.

## Bridge usability bugs discovered

All silent-failure-mode bugs. Tracked for a single follow-up commit, not yet
written:

1. **`-c <path>` is not parsed.** `main.c:32-33` takes `argv[1]` as the config
   path raw, so `delta-bridge -c foo.conf` reads `"-c"` as the path. Fix: real
   argument parser (`-c <path>`, positional, `-h`, reject unknown flags).
2. **`config_load` returns 0 ("success") when every key in the file is
   unknown.** Result: typos like `mqtt_host` (correct is `broker_host`) silently
   fall back to localhost. Fix: distinguish ENOENT from "file exists but parsed
   to nothing useful", warn on unknown keys.
3. **No startup log line shows the loaded config**, so there's no way to tell
   defaults from file values at runtime. Fix: log `broker=… user=… device_id=…
   topic_prefix=… poll=NHz` once at start.

The cumulative effect: a fresh user with a wrong config sees "broker down,
retry in 1s" forever and has no way to know it's a config typo.

## Tooling added this session

`tools/shmem_dump.c` — read-only diagnostic, cross-compiled for armv5te
static-pie via the same Docker workflow as `delta-bridge`. Modes:

```
shmem_dump                          one-shot named dump + hex windows
shmem_dump <seconds>                watch 0..0x2000 at 10 Hz; stdin Enter
                                    injects inline MARK lines for correlating
                                    physical events to byte changes
shmem_dump --save <path>            snapshot full 256 KiB to file
shmem_dump --diff <f1> <f2> [...]   diff 2+ snapshots side-by-side; any
                                    byte taking distinct values across the
                                    columns is a state-tracking byte
shmem_dump --help                   usage
```

The snapshot/diff workflow is the right tool to find cable-state under a
clean run — take a snap per physical state, diff, and the bytes that vary
across snapshots are the state-tracking bytes regardless of where they live in
the 256 KiB. We didn't get to use it productively in this session because
the undervoltage fault disables CP altogether.

Not committed yet; lives at `boards/eluminocity-ch21130/companion/tools/shmem_dump.c`.
